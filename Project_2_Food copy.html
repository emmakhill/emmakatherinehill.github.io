Group_by() and Summarize():

Summarize()

To start, we have the summarize() function, which is great for–you guessed it–reducing/collapsing the rows in our data to calculate very useful summary statistics. Some common summary functions one may use inside of summarize() include mean(), sum(), sd() (aka standard deviation), max(), and more! As an important note, the use of summarize() will produce a new summary data frame, which exists independently of your original data frame and will only contain the columns for each summary statistic you calculated within summarize(). To make this new summary data frame permanent, simply assign a object name to the pipeline using “=” or “<-”. In addition, to be safe, its best to add a “na.rm = TRUE” within your argument, so that any NAs that may exist in your data set are removed from the calculation. Within summarize(), the basic argument looks as follows: summarize([new variable name] = summary_function([columns to summarize], na.rm = TRUE))

Group_by()

Next we have the group_by() function, which is almost always used in combination with summarize(), thus making them a common pair! The group_by() function works by aggregating our data into groups based on the levels or categories contained within one or more specific columns, thus allowing us to perform calculations per group instead of across the entire data set. The argument passed within the group_by() function is whichever column(s)/variable(s) you wish to group by according to their levels. You can also group_by multiple columns, however the order matters, as your grouping will occur according to the first column entered, then this grouping will be grouped by the second column entered, and so on. group_by([column1, column2, …])

** Note: without group_by(), summarize() will only return one row for the entire data set, while when used with group_by(), it will return a row for each group. **

Now that we are beginning to understand group_by() and summarize(), lets look at an example of using these two functions within our food_consumption data. These functions can be great for some initial exploration of your data set, especially if your data set is quite large and contains a significant number of observations (rows) or variables (columns). Lets use them to produce a table showing the total_product_emission per each food product category.

food_consumption %>% 
  group_by(food_category) %>% 
  summarize(total_product_emission = sum(co2_emmission))

Mutate():

Mutate is a great function if you want to add a new variable to your data set (as a new column) using existing columns! In comparison to summarize(), mutate() leaves all existing rows and columns in your data frame and simply adds on a new column to the end. Within mutate, you can pass a Boolean argument or perform a calculation across variables. Within mutate(), your argument should first include the name of your new variable, then the assignment operator, then your calculation/manipulation. See below: mutate([new_column_name] = …)

Say we want to create a new variable based on whether the “food_product” type is an animal product or non-animal product. This example would require the use Boolean logic–if [this], then “animal_product”, else “non-animal product”–as the argument. We will specifically use an “ifelse()” statement, with the syntax as follows: “ifelse([test], [if true], [if false])”.

food_consumption %>% 
  mutate(Type = ifelse(food_category %in% c("Beef","Milk - inc. cheese","Lamb & Goat","Pork", "Fish", "Poultry", "Eggs"), "Animal_Product", "NonAnimal_Product"))

Left_Join:

Next we are going to learn about left_join(), which exists in the family of verbs called mutating joins within dplyr. These verbs allow you to add new variables (from one data frame) to another data frame based on matching observations. This idea of matching observations can also be referred to as using a matching “key”. A “key” is a variable/column that exists in both data frames you wish to join and can be used to match the observations from one to the other. Lets look at the syntax for left_join(): TableA %>% left_join(TableB, by = “Key”)

Left_join joins matching rows from TableB to TableA (the left-most table), such that all rows in TableA remain, while only those rows from TableB that match with TableA are joined–more simply, you are just adding on to TableA with that from TableB that matches.

At this current time, we have two independent data frames with useful information. “Type_total” contains the total emission and total consumption per product type for (an average person in) each country, and food_total contains the total emission and total consumption for all foods (consumed by an average person) in each country. My ultimate goal with this data set is that I want to understand how much of a role animal products vs non-animal products play in total food carbon emission rates per individual; in this way, I will be examine the role our food category choices play in the amount of co2 each individual emits into the atmosphere. Thus, I want the data for type_totals and food_totals in the same data frame, which we can accomplish this by using the left_join() function!

type_food_joined = type_total %>% 
  left_join(food_total, by = "country")

Explanation: As I want to keep all the rows in my type_total data frame, while simply adding on the additional variables that exist in my food_total data frame, I used “type_total” as my “TableA” and “food_total” as my “TableB”. Finally, as country is a variable that exists across both data frames, I used it as my key to match the two tables together by shared observations. I then assigned this joining pipeline to the new data table named “type_food_joined”.

Filter():

Filter() is a great function when you want to return only a select subset of rows from your data set. For this function, you simply identify the criteria–specifically, the types of values within a certain variable–in which you would like your rows to be filtered by. It is common to use comparison/logical operators within Filter(), such as > (greater than), < (less than), >= (greater than or equal to), <= (less than or equal to), == (equal to), != (not equal to). In addition, when using logical operators and Boolean statements, it is important to know that the operator for “and” is “&”, and the operator for “or” is “|”. The basic syntax for filter is as follows: filter([variable name] >/</=…)

Looking ahead to the next step of graphing, because this data set contains information for a whooping 130 countries total, I want to scale down my data set to a smaller, more meaningful size. Specifically, I would like to visualize the countries with the highest total_food_emission rates per person annually and the countries with the lowest rates. After examining my data, I decide to take the highest 6 and lowest 6 countries for their total_food_emission values. The highest 6 can be filtered by taking the rows with values more than 1700, and the lowest 6 can be filtered by taking the rows with values lower than 225.

type_food_joined %>%

Explanation: Were you thinking of the right logical operator? This pipeline is tricky because you are trying to return a table with rows greater than 1700 and rows less than 225–did the “and” mess you up?–lets think more deeply about the two possible operators. When using “&”, this means that all criteria must true for the argument to be true-thus, in this example, if we were to use and–“total_food_emission > 1700 & total_food_emission < 225”–that would mean that we would only be filtering by rows that meet both of those two criteria, which return zero rows. However, when using |–“total_food_emission > 1700 | total_food_emission < 225”–we are telling R to filter by any rows that meet this or that, thus returning all the “this” rows AND all the “that” rows.

To wrap up filter(), lets assign this pipeline to a new table named “type_food_filtered” to use when plotting:

type_food_filtered = type_food_joined %>% 
  filter(total_food_emission > 1700 | total_food_emission < 225)

Now lets plot our final wrangled data frame! Fct_reorder():

As you can see in the code below, I also used “fct_reorder” within my aesthetic plotting, specifically in regards to my x-axis aesthetic. Fct_reorder() is a handy function within the forcats package which allows you reorder factor/categorical levels by sorting along the values of another variable. This can be particularly handy as character categorical levels, such as country, are automatically arranged along the x-axis alphabetically. Since I instead want the countries to be arranged in descending order according to type_total_emission, this is what I will include within the fct_reorder() argument. Finally, map “Type” onto the “fill” aesthetic to show the proportion of animal products vs non-animal products within the total emission amount (as a stacked bar graph), adjust the theme and add some labels and Voila!

Plot Code: Total Kg

type_food_filtered %>% 
  ggplot(aes(x = fct_reorder(country, type_total_emission, .desc = TRUE), y = type_total_emission)) +
  geom_col(aes(fill = Type)) +
  theme(axis.text.x = element_text(angle = 90, hjust = 1, vjust = .5),
        legend.text = element_text(size = 7)) +
  labs(title = "Food Carbon Footprint per Person Annually",
       subtitle = "6 Highest and 6 Lowest Countries Worldwide", 
         x = "",
         y = "Total Emission (Kg co2/person/year)",
       fill = "Food Category") +
  scale_fill_manual(labels = c("Animal Products", "Non-Animal Products"), 
                      values = c("brown3", "olivedrab3"))

Through examining the two graphs we made, a few major findings become clear. Looking at our first graph, the discrepancy between the highest countries for food co2 emissions per person annually and the bottom counties is shocking, with almost 1500 more kg of co2 being emitted by the top countries. Another interesting finding is the difference between the highest emitting countries and the lowest emitting countries, that being that the highest 6 are composed mainly of highly developed, western nations–including the USA, ranking at the 6th highest–while the lowest 6 are composed mainly of developing or third-world nations. Liberia particularly stands out as the only country by which the average diet is composed more so of non-animal products composing as the majority. Our second graph allows us to look more closely at the proportion of the type of food consumed within each of these 12 nations. As it becomes clear, while the western nations (being the top emitting of co2) have diets composed of only around 25% non-animal products, the third-world developing nations with the lowest food co2 emission rates have diets in which non-animal products make up around 50% or more of their total food consumed.

Compared to nations with vastly less food-related co2 emissions, the western world consumes around 25% or more animal products.
